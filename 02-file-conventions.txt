FILE CONVENTIONS

Your operator model needs to live somewhere durable that every agent can find. This chapter covers the specific files you'll create — where they go, what goes in each one, and how to keep them clean. These files are the storage layer for your model.

THE FILES YOU'LL CREATE

  AGENTS.md in your canonical repo — your global rules
  AGENTS.md at ~/ — your machine's entry point, points to the global file
  AGENTS.md in any project repo — that project's specific guidance
  CLAUDE.md in any repo — instructions for Claude Code specifically
  GEMINI.md in any repo — instructions for Gemini specifically
  .github/copilot-instructions.md — instructions for Copilot
  .cursor/rules/*.mdc — instructions for Cursor
  agents.txt in any repo — one-line pointer to the entry file

You don't need all of these on day one. Start with the global AGENTS.md and ~/AGENTS.md. Add the rest as you need them.

WHAT GOES IN EACH FILE

  Global AGENTS.md: how you want agents to behave, your working rules, session protocols, communication preferences. Everything that applies across all projects.
  Machine AGENTS.md: pointer to the global file, your OS, installed tools, repo locations, running services. Facts about this computer.
  Repo AGENTS.md: build/test/deploy commands, architecture notes, file layout, project-specific rules. Facts about this project.
  Tool configs: anything that only matters for one specific AI tool.

KEEP THEM SEPARATE

  Rules and preferences go in AGENTS.md files.
  Machine facts go in the machine AGENTS.md.
  Tasks and to-dos go in whatever task system you use (or don't — that's optional).

Don't mix these. An AGENTS.md stuffed with task lists will rot. A task tracker stuffed with coding conventions will be ignored.

YOU DON'T HAVE TO WRITE THEM ALL YOURSELF

Your agent can help you draft these files. Ask it: "Help me create an AGENTS.md for this machine" or "Help me write an AGENTS.md for this project." You provide the details — what OS you're on, what the project does, how it builds — and the agent structures it into the right format. You review and refine from there.

Some tools can also inspect your environment (installed runtimes, directory structure, build configs) to suggest content. Whether that works depends on your tool's capabilities and permissions — it's a shortcut, not something the system depends on.
