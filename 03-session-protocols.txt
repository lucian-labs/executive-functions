SESSION PROTOCOLS

Every time you open an AI agent, it starts cold. These protocols fix that by defining what happens at the start, during, and end of each session.

WHEN A SESSION STARTS

Put a checklist in your global AGENTS.md that your agent runs on startup:

  1. Read the instruction chain: global AGENTS.md → machine AGENTS.md → repo AGENTS.md → tool config
  2. Check for any queued work (if you use a queue system — see 05)
  3. Look at git status and check for in-progress work from last time
  4. Verify any local services the project needs are running

By the time you give your first instruction, the agent already has full context. No re-explaining.

If you run a coordination system (dashboard, API, shared queue), the agent can register on startup to announce itself and pick up pending work. This is optional — the system works fine without it.

DURING A SESSION

Your agent should:

  Execute your instructions immediately without asking unnecessary questions
  Only pause for decisions that are irreversible or high-stakes
  Update context files when it discovers new information about your setup
  Log significant events (features shipped, decisions made)

If you run multiple agents at once (different tools, different terminals), they should stay in their lane: ignore changes they didn't make, don't revert someone else's work, only flag conflicts if they hit the exact same files.

WHEN A SESSION ENDS

Whether you close the chat, the agent hits its context limit, or the work is done:

  1. Update your context files with anything new the agent learned
  2. Write a short summary of what happened (for handoff to the next session)
  3. If ending mid-task, include enough context that the next session can pick up where this one left off

The key principle: knowledge flows outward. Anything important the agent learned should end up in a file, not trapped in a chat transcript you'll never read again.
