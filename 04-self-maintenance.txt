AGENT SELF-MAINTENANCE

Your operator model isn't static — it gets sharper every session. The trick: your agents are responsible for updating the files that define your model. When they learn something new about your workflow, they write it back. This is what makes the system compound instead of decay.

THE PROBLEM YOU'VE PROBABLY HIT

You write a great CLAUDE.md. It works perfectly for two weeks. Then you add a new build step, change a directory structure, or start using a new tool — and the file is stale. You don't update it because you're busy doing actual work. The agent starts making mistakes based on outdated context.

THE FIX

Tell your agents (in the global AGENTS.md) that they're responsible for updating the files they read. When an agent discovers something new during a session — a file moved, a build command changed, a new convention — it writes that into the appropriate context file before the session ends.

Next session, the updated context is already there. No re-explaining. The system gets smarter with every use.

HOW TO SET THIS UP

Add something like this to your global AGENTS.md:

  "When you discover new patterns, file locations, or constraints while working, add them to the appropriate agent files. Don't leave knowledge only in the conversation."

Then specify what goes where:

  Repo-level discoveries (build changes, new files, architecture notes) → that repo's AGENTS.md
  Machine-level discoveries (new services, path changes, installed tools) → ~/AGENTS.md
  Task updates (completed, blocked, deferred) → your task system if you have one

RULES TO INCLUDE

  1. Knowledge flows outward — from conversation into files, not the other way around
  2. Stay in scope — an agent working on frontend shouldn't edit backend docs
  3. Add, don't delete — new information gets appended. Only remove things that are provably wrong
  4. Machine-specific stays local — absolute paths and ports never go in committed files
  5. Don't narrate it — self-maintenance is background work, not something to report on

THE BOOTSTRAP

First time on a new machine or project, there are no files to read. The agent creates them by scanning what's there: filesystem, installed tools, READMEs, build configs. After that first session, every subsequent one starts faster. It compounds.

WATCH OUT FOR

  Hoarding: agent discovers something important but only mentions it in chat. Next session can't see it.
  Over-updating: writing every trivial observation into context files. Keep it to durable, reusable knowledge.
  Cross-contamination: project rules leaking into the global file, or machine paths leaking into repo files.
