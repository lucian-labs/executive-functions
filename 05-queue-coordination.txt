QUEUE COORDINATION

This chapter is optional. If you're running one agent on one machine, skip it. Come back when you need multiple agents to hand work to each other.

WHAT QUEUES DO

A queue is a list of work orders. You (or another agent) push a message to a named queue. The agent listening on that queue picks it up and executes it.

  You push "deploy the staging build" to the deploy queue
  The agent on your server picks it up and runs the deploy
  It reports back what happened

WHEN YOU NEED THIS

  You use multiple AI tools and want them to coordinate
  You work across machines (laptop + server, laptop + Raspberry Pi)
  You want to queue up work for an agent to pick up later

MESSAGE TYPES

Keep it simple. Three types cover most needs:

  feature — build this thing
  bug — fix this thing
  todo — look into this and create a tracked task

Messages are directives, not suggestions. When one arrives, the agent acts on it.

HOW TO NAME YOUR QUEUES

Pick a scheme that matches how you think about your work:

  By project: my-app, my-website, my-plugin
  By category: frontend, backend, infra, audio
  By machine: laptop, server, pi
  By role: deploy, review, research

There's no enforced convention. Pick what makes sense to you.

IF A MESSAGE CAN'T BE HANDLED

Sometimes the current session can't finish a work order (too big, blocked, wrong machine). When that happens: create a tracked task with the full context, resolve the original queue message. Don't leave stale messages sitting around.

HOW TO IMPLEMENT

Start as simple as you need:

  Simplest: a queue/ folder with one file per message. Agent checks on startup. Done items move to queue/done/.
  Middle: a small server with REST endpoints for push and poll.
  Full: a dashboard with a web UI, notifications, session tracking.

The patterns in this chapter work the same at every scale. Start with folders, upgrade if you need visibility.
