MULTI-MACHINE SCALING

Optional. Your operator model works on one machine by default. This chapter covers how to carry it across multiple machines so every agent, everywhere, follows the same model.

THE SETUP

Your global AGENTS.md lives in a git repo. Every machine you work on clones that repo and creates a local ~/AGENTS.md that points to it. When you update the global file and push, every machine picks up the changes on next pull.

  Your canonical repo (AGENTS.md)
    Laptop ~/AGENTS.md → points here
    Desktop ~/AGENTS.md → points here
    Server ~/AGENTS.md → points here
    Pi ~/AGENTS.md → points here

WHAT STAYS LOCAL

Each machine has its own context that doesn't get shared:
  OS and platform
  Installed runtimes and versions
  Absolute paths (different on every machine)
  Running services and ports
  Connected hardware (audio interfaces, MIDI controllers, displays)

This goes in each machine's ~/AGENTS.md, not in the canonical repo. Universal rules live in the global file; local facts live on the machine.

COORDINATING ACROSS MACHINES

If you need agents on different machines to pass work to each other, use the queue system from chapter 05. A message pushed to a queue gets picked up by whichever machine's agent is listening.

Examples:
  Your laptop pushes code → your server's agent deploys it
  Your laptop queues a heavy build → your desktop (more powerful) runs it
  Any agent queues a hardware command → your Pi (connected to the device) sends it

FILE TRANSFER

For moving files between machines without committing them to git, use whatever shared storage you already have — cloud drive, S3 bucket, simple upload/download endpoint, even scp. Useful for build artifacts, screenshots, config files during setup.

ADDING A NEW MACHINE

  1. Clone your canonical repo
  2. Create ~/AGENTS.md pointing to the global file
  3. Fill in the local details: OS, installed tools, repo locations, running services
  4. (Optional) Set up credentials for your coordination system if you use one

You can ask your agent to help draft the machine file — "Help me create a machine AGENTS.md for this computer" — and fill in what it can't determine on its own.

WHY LOCAL CONTEXT DOESN'T SYNC

By design. Each machine's facts are local because they're different on every machine. If you want machines to know about each other, that's what the coordination system (queues, dashboard) is for.

The instruction hierarchy, session protocols, and memory alignment work identically everywhere. The only thing that changes per machine is the local facts.
